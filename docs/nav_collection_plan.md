# 基金净值历史数据采集方案

## 一、目标

为规模≥2亿且已有最新净值的 **4,467 只基金** 补采完整历史净值数据（从成立日期到最近日期）。

---

## 二、数据现状

| 类别 | 数量 | 数据完整度 |
|------|------|------------|
| 已有完整历史 (≥1000条) | ~12只 | 接近100% |
| 部分历史 (100-1000条) | ~30只 | 部分缺失 |
| 仅最新净值 (≤7天) | 4,453只 | 仅最近几天 |
| 无净值数据 | 655只 | QDII/定期开放基金 |

---

## 三、技术方案

### 3.1 数据源

- **接口**: `akshare.fund_etf_fund_info_em(fund=code)`
- **返回**: 基金净值历史（单位净值、累计净值、日增长率）
- **限制**: QDII基金、定开基金可能无数据

### 3.2 采集速度估算

- 单基金平均耗时: ~3秒（含网络延迟）
- 4467只基金理论耗时: **约3.7小时**
- 考虑失败重试、网络波动: **约5-6小时**

## 四、开发任务

### 4.1 主采集脚本

```python
# scripts/collect_nav_history.py

功能:
- 查询需要补采历史的基金列表
- 遍历调用 AkShare 接口
- 存储到 fund_nav 表
- 记录采集日志和进度

参数:
- --batch-size: 每批处理数量 (默认100)
- --delay: 请求间隔秒 (默认0.5)
- --retry: 失败重试次数 (默认3)
- --log-file: 日志文件路径

输出:
- 采集进度 (当前/总数)
- 成功/失败统计
- 错误明细
```

### 4.2 断点续采机制

```python
# 断点记录表: nav_collection_status
- fund_code: 基金代码
- status: pending/processing/completed/failed
- records_count: 已采集记录数
- last_attempt: 最后尝试时间
- error_msg: 错误信息
```

### 4.3 失败重试脚本

```bash
# 重新采集失败的基金
python scripts/retry_failed.py --max-retries 3
```

### 4.4 定时任务配置

```bash
# 方案1: 手动运行
python scripts/collect_nav_history.py

# 方案2: 定时任务 (建议晚上执行)
0 21 * * * cd /path/to/fund-system && python scripts/collect_nav_history.py >> /var/log/nav_collection.log 2>&1
```

## 六、风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| AkShare接口限速 | 采集慢 | 增加请求间隔 |
| 网络中断 | 采集中断 | 断点续采 |
| QDII基金无数据 | 655只无法采集 | 记录并跳过 |
| 数据库锁 | 写入失败 | 事务+重试 |

2. **数据量**: 目标总记录数≥300万条
3. **时间范围**: 数据覆盖成立日期至今
