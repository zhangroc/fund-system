# 基金净值数据验证方案

## 一、验证目标

确保采集的净值数据**准确、完整、合理**，满足回测分析需求。

---

## 二、验证维度

### 2.1 覆盖率验证

**目标**: 4,467只基金中尽可能多的基金有完整历史数据

| 指标 | 合格标准 |
|------|----------|
| 有历史数据的基金比例 | ≥80% |
| 有≥1年数据的基金比例 | ≥50% |
| 有完整历史(成立至今)比例 | ≥30% |

**SQL查询**:
```sql
SELECT 
  COUNT(*) as total,
  SUM(CASE WHEN nav_days >= 365 THEN 1 ELSE 0 END) as one_year,
  SUM(CASE WHEN nav_days >= 730 THEN 1 ELSE 0 END) as two_years
FROM (
  SELECT fund_code, COUNT(DISTINCT DATE(nav_date)) as nav_days
  FROM fund_nav
  WHERE fund_code IN (SELECT fund_code FROM funds WHERE scale >= 2)
  GROUP BY fund_code
)
```

---

### 2.2 数据完整性验证

**目标**: 检查净值记录的日期连续性和数据完整性

| 检查项 | 验证逻辑 | 合格标准 |
|--------|----------|----------|
| 日期连续性 | 相邻日期差距≤7天 | 90%日期有数据 |
| 字段完整性 | nav/accumulated_nav非空 | 100% |
| 重复检查 | 同日同基金多条记录 | 0条重复 |

**SQL查询**:
```sql
-- 检查重复记录
SELECT fund_code, nav_date, COUNT(*) as cnt
FROM fund_nav
GROUP BY fund_code, DATE(nav_date)
HAVING COUNT(*) > 1;

-- 检查字段完整性
SELECT COUNT(*) as missing_nav FROM fund_nav WHERE nav IS NULL;
```

### 2.3 数据准确性验证（关键）

**目标**: 与数据源（天天基金网）对比，确保数值准确

**验证方法**: 随机抽取10只基金，对比以下字段：

| 字段 | 验证方式 |
|------|----------|
| 单位净值 | 与官网对比，误差=0 |
| 累计净值 | 与官网对比，误差=0 |
| 日增长率 | 与官网对比，误差≤0.01% |

**抽样基金列表**:
```
000001 平安新利混合
000011 华夏大盘精选混合A
000015 华夏纯债债券A
000009 易方达天天理财货币A
000076 华夏恒生ETF联接
```

**验证脚本**:
```python
# scripts/validate_accuracy.py
```

### 2.4 数值合理性验证

**目标**: 检查净值数据是否在合理范围内

| 检查项 | 验证逻辑 | 异常标准 |
|--------|----------|----------|
| 净值范围 | 单位净值 > 0 | ≤0 异常 |
| 日增长率 | -15% ≤ 增长率 ≤ +15% | 超出异常 |
| 净值连续性 | 累计净值 ≥ 单位净值 | 违反异常 |
| 货币基金 | 日增长率在 0.001%~0.2% | 货币基金异常 |

**SQL查询**:
```sql
-- 检查异常净值
SELECT fund_code, nav_date, nav, accumulated_nav
FROM fund_nav 
WHERE nav <= 0 OR accumulated_nav < nav;

-- 检查异常增长率
SELECT fund_code, nav_date, daily_growth
FROM fund_nav 
WHERE daily_growth < -15 OR daily_growth > 15;
```

### 2.5 与历史回测对比

**目标**: 确保历史回测结果与之前一致

| 检查项 | 验证方法 |
|--------|----------|
| 已有回测 | 重新运行历史回测，对比收益曲线 |
| 基准对比 | 与沪深300指数对比 |
| 极值检验 | 模拟极端市场情况 |

## 三、验证流程

```
┌─────────────────────────────────────────────────────────────┐
│                        验证流程                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Step 1: 覆盖率检查                                          │
│    └> 查询统计: 有数据基金数、历史深度分布                    │
│    └> 不合格? -> 返回采集阶段                                 │
│                                                             │
│  Step 2: 完整性检查                                          │
│    └> SQL检查: 重复、缺失、连续性                            │
│    └> 不合格? -> 数据清洗/补采                               │
│                                                             │
│  Step 3: 准确性检查                                          │
│    └> 抽样对比: 10只基金 vs 官网                            │
│    └> 不合格? -> 定位问题、修复                               │
│                                                             │
│  Step 4: 合理性检查                                          │
│    └> SQL检查: 异常值、边界情况                              │
│    └> 不合格? -> 数据清洗                                   │
│                                                             │
│  Step 5: 回测验证                                            │
│    └> 重新运行历史回测                                      │
│    └> 对比结果一致性                                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 六、问题处理

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 部分基金无历史 | QDII/定开基金 | 记录并跳过 |
| 净值与官网不符 | 数据源问题 | 重新采集 |
| 重复记录 | 采集逻辑问题 | 去重清洗 |
| 增长率异常 | 数据源问题 | 标记或修正 |
